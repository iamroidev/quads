### Recommended Additions

#### High Value, Low Effort

1. **Redis (Upstash / ElastiCache)** — *Cache & Rate Limiting*
   - **Why:** Collections API (`aiCollectionsCache`) uses in-memory caching (lost on restart). Redis gives a shared TTL cache across server instances, survives restarts, and fixes rate-limit middleware which currently uses an in-memory Map.
   - **Effort:** Low. Add `ioredis` or `@upstash/redis`, swap 3-4 caches + rate-limit store.

2. **Sentry** — *Error Monitoring*
   - **Why:** Error handling is `try/catch → console.error` everywhere. Sentry captures unhandled rejections, gives stack traces, release tracking, and alerting.
   - **Effort:** Low. One SDK install, one `init()` in `app.ts`, environment-based sample rates.

3. **AWS S3 (or Cloudflare R2)** — *File Backup / Fallback*
   - **Why:** Cloudinary is the primary upload target, but there is zero fallback. Having S3 as a secondary or backup store prevents full upload outage if Cloudinary is unavailable. Already partially wired in `imageUpload.ts` for a local filesystem fallback — S3 is the natural extension.
   - **Effort:** Medium. SDK install, add S3 upload path in `uploadToCloudinary` / `uploadMultipleWithFallback`.

#### Medium Value, Medium Effort

4. **Coravel / BullMQ (Redis Queue)** — *Job Scheduler*
   - **Why:** The auto-payout system runs every 15 minutes via a raw `setInterval` inside `payoutScheduler.ts`. Background jobs (payouts, notifications, import retries) should be properly queued with retries, dead-letter queues, and persistence. The `RetryJob` model already exists — BullMQ is a natural fit.
   - **Effort:** Medium. Replace `setInterval` in `payoutScheduler` with a recurring BullMQ job. Wire `RetryJob` model as queue state.

5. **Twilio (or AWS SNS fully configured)** — *SMS*
   - **Why:** SNS code is ready but production SMS relies on AWS account credentials. Twilio has a simpler developer experience, better delivery receipts, and sandbox mode for testing.
   - **Effort:** Low-Medium. Swap the SNS `PublishCommand` call in `verification.service.ts` for a Twilio `messages.create` call.

6. **Cloudflare** — *CDN + WAF + DDoS Protection*
   - **Why:** Frontend is presumably on Vercel (client_url = `quadsmarket.tech` / VITE env), but API is on an EC2 server. Cloudflare in front of the EC2 instance gives free CDN, bot protection, rate limiting, and SSL. Product images from Cloudinary already benefit from their CDN, but every HTML/JS payload hits origin directly.
   - **Effort:** Low (DNS + tunnel/proxy).

7. **PostHog or Plausible** — *Product Analytics*
   - **Why:** `AnalyticsEvent` model and `captureEvent()` exist in the codebase but there is no frontend instrumentation that emits real `view` / `chat` / `order` events at scale. PostHog/Plausible provides session replay, funnel analysis, and event dashboards without GDPR headaches.
   - **Effort:** Low. Embed PostHog snippet in `index.html`, add React wrapper for `posthog.capture()`.

#### High Value, Higher Effort

8. **OpenAI / Anthropic** — *AI Enhancements*
   - **Why:** `ai.service.ts` already exists and does product grouping for collections. Adding LLM capabilities enables: smarter listing descriptions (auto-generate from title), auto-categorization, dispute summarization for admins, fraud detection patterns, and AI-assisted trust-signal scoring.
   - **Effort:** Medium-High. Needs prompt engineering, moderation guardrails, cost controls.

9. **Helicone / Langfuse** — *LLM Observability*
   - **Why:** If AI features are added (above), tracing LLM calls is essential for cost tracking, latency analysis, and prompt quality.
   - **Effort:** Low (if paired with #8).

## Part 2: Feature Ideas & Gaps

### Features Ready to Build

Judging by what's already scaffolded, typed, and wired:

**Buyer-Facing**
- Saved items collections / wishlists — UI exists (`SavedItems.tsx`) and service is ready. Could benefit from public shareable lists.
- Order tracking timeline — `OrderDetail.tsx` shows status but lacks a visual timeline/progress bar.
- Bulk price-drop alerts — The `ProductService.updateProduct` sends notifications when price drops for saved-item users. Expandable to a "watch this item" feature that alerts on any status/reprice change.
- In-app notifications panel — `Notifications.tsx` exists; expand to include in-app toasts for message/review/order events (currently just a list).
- Proximity-based discovery — `pickupLocation` filter exists in product queries; a "near me" map view is not implemented.
- Academic calendar integration — The marketplace is UMaT-specific. Adding a semester-aware "textbook season" or "fresher drop" filter by academic period aligns with the institution.

**Seller-Facing**
- Webhook / Zapier endpoint for seller shops — Payout completion, new order, low-stock webhooks would let sellers hook into their own tools.
- Inventory low-stock alerts — `Product.stock` exists but `Dashboard` doesn't show low-stock items. Add a `stock <= threshold` check in Seller Analytics.
- EPUB/PDF preview for textbook listings — Categories already include `textbooks`. A text-preview or condition notes field would increase buyer confidence.
- Seller profile public page — Currently sellers are view-able at `/sellers/:id` (redirects to product filter `?seller=`). A dedicated seller storefront with branding, ratings badge, and all active listings is a stub — could be a real page.
- Multi-variant products — The `Product` model has `stock` and `price` but no `variants` array (size, color, bundle options). Adding variant support for clothing/electronics is high-value.
- Commission rate display — Sellers don't see their platform commission % on payout pages. Adding the explicit commission rate (currently hidden in `Payout`) builds trust.
- Photo gallery editor — Product detail images can be uploaded but re-ordering and deleting individual images mid-edit is not implemented in `CreateEditProduct.tsx`.

**Platform / Admin**
- Admin product importer moderation queue — The CSV import bypasses review. An admin "review imported items" queue prevents bad bulk imports hitting the marketplace.
- User activity timeline — A full activity log per user (logins, listing creates, orders, verification events) in `OpsAuditLog` — admins can see per-user audit trails.
- Scheduled content management — The `content/category-spotlights` exists but has no CMS interface. A rich-text / markdown editor updating `Category.description` or adding manual spotlights.
- Scheduled maintenance mode — `Maintenance.tsx` exists, but there is no backend toggle to flip the app to 503. Should be an admin-only env/config toggle.
- Stripe / Flutterwave alternative payments — Currently only Paystack. Adding a secondary provider (Flutterwave) for users who don't have Paystack accounts increases coverage.
- Referral program — Referral codes per user, referral rewards (discount or credit), `referrer` field on `User` model.
- Seasonal promotions — Integration with the growth campaign system for "back to school," "graduation sale," "freshers week" campaigns that auto-active in date windows.

## Part 3: Unfinished / Hanging Features

### In Code but Not Functional

#### 1. Password Reset Flow — Page exists, backend missing
- **`ForgotPassword.tsx`** and **`ResetPassword.tsx`** routes exist in `App.tsx`
- **`ForgotPasswordPage`** renders UI for email input
- **`ResetPasswordPage`** renders token + new-password form
- **`auth.controller.ts`** has **no** `forgotPassword` or `resetPassword` endpoints
- **`auth.service.ts`** has **no** forgot/reset logic
- `NEXT_STEPS.md` does not reference this gap
- **Impact:** Users who lose access to their account (forgot password + Supabase is the true auth, mpassword is a random fallback per `User` model) have no recovery path via the platform. They must rely entirely on Supabase's own password reset flow.

#### 2. Firebase Phone Auth — Imported but not configurable
- `server/src/config/firebase.ts` is imported by `verification.service.ts` (`import firebaseAdmin from '../config/firebase'`)
- The `verifyFirebasePhone` method calls `firebaseAdmin.auth().verifyIdToken(idToken)`
- No evidence of `FIREBASE_ADMIN_SDK_KEY` or equivalent in `server/src/config/env.ts`
- If `firebaseAdmin` is `null`, the method throws `ApiError.internal` at runtime
- **Impact:** Phone verification via Firebase is dead code until Firebase Admin credentials are added to environment config.

#### 3. AWS SNS SMS — Same credential gap as Firebase
- `sns = new SNSClient(snsConfig)` in `verification.service.ts` uses `env.AWS_ACCESS_KEY_ID`
- If `AWS_ACCESS_KEY_ID` is absent, falls through to `{ region: env.AWS_REGION }` — the SDK's default provider chain may or may not resolve credentials in the EC2/PM2 production context
- If AWS credentials are not provisioned, SMS OTP silently fails
- **Impact:** Unverified on production. Users relying on SMS verification may not receive codes.

#### 4. Scholar Badge — Concept exists, not implemented
- Mentioned in `NEXT_STEPS.md` as a Phase 2 item: *"Implement the 'Scholar Badge' logic for students who have verified their institutional ID"*
- `User.isInstitutional` exists in the model (set from `INSTITUTIONAL_DOMAINS` in env)
- No logic ties `isInstitutional` to any visible badge, profile display, or special listing privileges
- **Impact:** `isInstitutional` field is set but serves no purpose in the UI.

#### 5. In-App Push Notifications Setting — Toggle exists, delivery unverified
- `Settings.tsx` has a push toggle; frontend calls `notificationService.subscribeToPush()`
- `notification.service.ts` uses `VAPID_PUBLIC_KEY` from env to subscribe via `PushManager`
- Backend has `POST /notifications/push/subscribe` (route exists in `notification.routes.ts` presumably)
- Backend push delivery uses `web-push` or `fcm-node` internally — needs VAPID key pair
- If VAPID key isn't set, push subscription throws and the user sees a toast error
- **Impact:** Push notifications toggle may break silently for users who don't have a valid VAPID key provisioned.

#### 6. Smart Pricing (SellerAnalytics + GrowthTools) — Dual copies
- Both `SellerAnalytics.tsx` and `GrowthTools.tsx` independently call coupon/bundle creation APIs
- They each call `orderService.getSellerCoupons()`, `orderService.getSellerBundles()`, `growthService.listCampaigns()` — no shared state
- The data mutation calls (`createCoupon`, `createBundle`, `createCampaign`) are in both files separately
- There is **no reactive refetch** across pages; flipping from Analytics → GrowthTools gives stale data unless user refreshes
- **Impact:** Duplicate CRUD logic, stale state, one coupon/edit won't show on the other page without hard refresh.

#### 7. Dispute Center — UI complete, seller-side workflow incomplete
- `DisputeCenter.tsx` (buyer view): complete
- `AdminDashboard.tsx` (admin moderation): complete
- **No seller-side dispute response view exists** — a seller cannot add evidence, reply, or see dispute status from their own dashboard
- **Impact:** Sellers are effectively silent in the dispute process, which slows resolution.

#### 8. Coupon Auto-Surfacing at Checkout — Partially done
- `NEXT_STEPS.md` lists: *"Automatic Discovery: In-page coupon surfacing for buyers (Amazon-style)"*
- `Coupon` model exists, `Order` row stores `couponCode`
- Checkout page (`Checkout.tsx`) applies coupon codes manually — no auto-suggestion loop
- **Impact:** Buyer doesn't see eligible coupons at the point of highest intent.

#### 9. Rotten/Hanging: Admin Growth Trust Signals Manually Entered  
- `AdminGrowth.tsx`: Admin enters `trustUserId` as a hard-coded string and `scoreDelta` manually
- There is no admin UI that shows a searchable list of users to select from — it's just a raw text input
- `getTrustSummary` exists but the admin page never calls it for display
- **Impact:** Trust scores are manually edited without context, no validation, and no audit trail for *why* a score changed other than the free-text `note` field.

#### 10. Auto-Payout Scheduler — Runs but unmonitored
- `autoPayout.service.ts` runs a `setInterval` that runs every `PAYOUT_INTERVAL` ms
- If the PM2 process restarts, the interval restarts — that's fine for `scheduleNextRun()`
- But there is no dashboard widget showing: last run timestamp, last run result, next scheduled run, queue depth
- Only `Payout` model / `/payouts/seller` and `/admin` payout tab show payouts retrospectively
- **Impact:** Ops team has no real-time visibility into whether the payout system is actually working.

#### 11. Incoming Routes Without Implemented Controllers

## Part 4: The Buyer/Seller Identity Inconsistency (Core Problem)

### What exists today
- `User._id` is simultaneously the buyer identity (`Order.buyer`) and seller identity (`Product.seller`, `Order.seller`)
- `switchRole` toggles the `role` field and **rotates the JWT** — meaning a seller who switches to buyer role has a token whose `role` claim is `buyer`, but the same `User._id` underneath
- Product ownership checks (`product.seller.toString() !== userId`) compare against `req.user._id`
- Seller routes (`/sell`, `/my-listings`) are guarded by `ProtectedRoute roles={['seller','admin']}`

### What breaks in practice
1. A seller switches to buyer mode → `role` becomes `'buyer'` in the JWT
2. `ProtectedRoute` now denies access to `/sell`, `/my-listings`, `/my-analytics` because the `role` check fails
3. Even if the route were accessible, the buyer-mode user misses their own products in `viewMode === 'buyer'` dashboards
4. No way to identify "this buyer is *also* a seller with listings" without re-switching roles

### Recommended Fix Phase (from the existing plan draft, Part A)

Add a `sellerId: ObjectId` field to `User`. First time a user activates seller mode, generate a new distinct ObjectId and store it. From there forward:
- `Product.seller` references `User.sellerId`
- `Order.seller` references `User.sellerId`
- `Payout.seller` references `User.sellerId`
- `Review.seller` references `User.sellerId`
- `Order.buyer` stays `User._id`
- The JWT carries both `userId` (buyer identity) and `sellerId`
- `role` stays `'seller'` for routeguard purposes; the frontend `viewMode` is a separate concern

This decouples "who is buying" from "who is selling" for the same person and eliminates the need for role switching entirely.
